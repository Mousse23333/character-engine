**任务对象**：运行在 128GB Blackwell 工作站上的 AI 助理（你）

## 0. 给执行者（PGX AI）的指令

### 0.1 你的角色
你是**独立调查员**，不是命令执行者。架构师不知道每个工具今天的真实状态、不知道每个模型在这台具体硬件上的真实表现。**你的任务是替架构师摸清楚**。

### 0.2 你享有的自由
- **不必按本文档章节顺序执行**。哪个方向先有信号就深入哪个
- **不必跑完所有任务**。优先跑最能产生"项目决策价值"的任务
- **可以放弃明显不可行的任务**，但必须在报告里说明放弃理由
- **可以追加本文档没列的实验**，只要你判断它对架构师有价值
- **可以在报告里挑战本文档的前提假设**

### 0.3 你必须做的
- 每个尝试都记录：**用了什么工具/模型/版本、参数、耗时、VRAM、产出质量**
- 失败比成功更重要 — 失败案例必须留底（截图/日志）
- 主观评级（业余可用 / 接近专家 / 不可用）必须配数字佐证
- **节省架构师时间是你的最高目标**：报告先给结论再给细节

### 0.4 资源约束
- GPU：128GB Blackwell（B200 级），可大胆并行
- 时间：约 8-12 小时（请把昂贵任务排在前半夜，便宜任务排在后半夜）
- 网络：可用，可下载任意开源模型 / clone 仓库 / 拉 docker
- 磁盘：假设充裕，但请清理超过 20GB 的临时产物
- 软件：可自由安装 conda env / docker / 任何开源工具
- 不可用：任何付费 SaaS（架构原则 AC-6 — 全开源）

### 0.5 命名约定
- 实验编号：`E-XX-描述`（如 `E-01-flux-baseline`）
- 产出物路径：`CharacterEngine/pgx_reports/<timestamp>/E-XX/`

## 3. 任务详细规格

### E-00: 数据扩充（前置任务）
**问题**：单张 189×267 参考图，怎么变成可训 LoRA 的 30-50 张数据？

**尝试方向**：
1. Real-ESRGAN / SwinIR 超分到 1024+
2. Flux Kontext / IP-Adapter 用单图作 condition 生成不同 pose / 角度
3. ControlNet OpenPose + ref-only 风格转移
4. 多次生成 → CLIP-I 自动筛 top-N

**报告关键点**：
- 哪种扩充方法 **最稳定**（CLIP-I 距离最小）
- 产出图能否承受人眼 "这就是同一个角色" 的判断
- 时间成本

---

### E-01: Flux + Illustrious 单图基线
**目的**：确认基础出图能力

**做什么**：
1. 部署 **Flux.1 dev**（或 Flux.2 如果开源版本可用）+ **Illustrious XL**
2. 用 prompt 描述测试角色，**不带任何 LoRA / IP-Adapter**
3. 各出 20 张
4. 比较：仅靠 prompt 能逼近多少？

**报告**：基线分（CLIP-I 与参考图距离）+ 主观评分

---

### E-02: Style/Identity 解耦（双 LoRA）
**目的**：验证架构师 AC-4（画风与角色解耦）的工程可行性

**做什么**：
1. **Style LoRA**：用 200+ 张同画风但**不同角色**的图训练（你需要找数据集，建议 Danbooru 标签筛选）
2. **Identity LoRA**：用 E-00 扩充出的同角色数据训练
3. 四组对比（每组 20 张）：
   - a) 仅 Style
   - b) 仅 Identity
   - c) Style + Identity 同时叠加（关键）
   - d) 不带 LoRA
4. 评判 c 是否同时呈现 a 的画风 + b 的角色身份，量化"互相污染"程度

**报告关键点**：
- 双 LoRA 叠加可行吗？污染严重吗？
- 如果污染严重，有什么缓解方法（权重调整 / LoRA 融合 / DARE-TIES 等）？
- 这条架构线（解耦）现在能不能走通？

---

### E-03: IP-Adapter / PuLID / InstantID 无训练对比
**目的**：评估"零训练"路线的真实质量

**做什么**：
1. 用 IP-Adapter / PuLID / InstantID 三个开源方案（仅二次元友好的）
2. 都用单参考图 → 出 20 张同角色不同姿态
3. 对比 vs 训练 LoRA 的质量差距

**报告**：何时该用零训练，何时必须训 LoRA

---

### E-04: Flux Kontext 多参考图一致性
**目的**：测试当前 SOTA 多参考图一致性技术上限

**做什么**：
1. 部署 Flux Kontext 开源版本（如果可用；否则替代品）
2. 喂 4-10 张 E-00 扩充图作为 reference set
3. 让模型出新姿态 / 新视角
4. 比较 vs LoRA 路线

**报告**：开源多参考图技术目前真实状态？

---

### E-05: Hunyuan3D 2.0 mesh 生成
**目的**：验证 H1 — Hunyuan3D 二次元角色 mesh 质量

**做什么**：
1. 部署 **Hunyuan3D 2.0**（Apache 2.0）
2. 输入：你 E-01-E-04 选出最好的角色正面图
3. 出 mesh + 贴图
4. 检查：
   - 三角面数（原始 / 简化后）
   - 拓扑质量（流形检测、自相交、edge flow）
   - UV 质量
   - 贴图质量（撕裂、走样）
   - **背面质量**（AI 3D 的硬伤）
5. 试试多视图模式（喂 4 张正交图）vs 单图模式，对比

**报告**：质量真实状态、failure modes、能否进入下游动画管线

---

### E-06: TRELLIS mesh 生成
**做什么**：同 E-05，但用 TRELLIS（如果开源可装）

**报告**：与 Hunyuan3D 2.0 横向对比，谁更适合二次元

---

### E-07: Retopology 开源方案评估
**目的**：AI 出的高密度 mesh 怎么变成动画可用拓扑

**做什么**：
1. 拿 E-05/E-06 的 mesh
2. 跑 **Instant Meshes**（开源）
3. 跑 **Blender QuadriFlow**（内建）
4. 跑其他开源 retop 方案（你调研找）
5. 对比：保形度、拓扑流向（特别是关节位置）、面数缩减比

**报告**：开源 retop 现在能不能用？如果不能，需要做什么补救

---

### E-08: 自动绑骨开源方案对比
**目的**：架构红旗确认 — 开源绑骨真实状态

**做什么**：
1. 拿 E-07 retop 后的 mesh
2. 试以下方案：
   - **UniRig**（SIGGRAPH 2025 开源）
   - **HumanRig** 或 **ASMR**（学术开源）
   - **Blender Rigify**（半自动）
   - **AccuRIG 2**（闭源但免费 — 仅作 baseline 比较，不进我们 pipeline）
3. 对比绑骨成功率、骨骼位置准确性、蒙皮权重质量
4. 用 Mixamo 标准动画驱动 → 看形变 artifact

**报告关键点**：
- 开源工程化的真实差距有多大（数字 + 主观评分）
- 哪条开源路线最接近可用
- 需要多少工程投入才能让某个开源方案 "能用"

---

### E-09: Mixamo 标准动画驱动
**目的**：验证绑好的角色能不能跑标准动画库

**做什么**：
1. 把 E-08 绑好的角色导出 FBX
2. 上传 Mixamo 套用 idle / walk / run / wave 4 个标准动画
3. 截屏 / 录视频
4. 检查穿帮（手指穿模、肩膀塌、髋关节扭曲等）

**报告**：FBX 工程链路通不通、典型穿帮模式

---

### E-10: 视频提取动作（DWpose / EasyMocap）
**目的**：动作来源的"AI 路径"可行性

**做什么**：
1. 找一段网络上的二次元舞蹈 / 走路视频（开源版权或自录）
2. 用 **DWpose** 提取 2D 关键点
3. 用 **EasyMocap** 提取 SMPL-X 3D 骨骼
4. 重定向到 E-08 绑好的角色
5. 看动画质量

**报告**：从视频到 rigged 动画的端到端工艺现在通不通

---

### E-11: Wan 2.2 角色 LoRA 训练 + 生成 ★
**目的**：验证 H2 — 视频生成模型 + 角色 LoRA 的稳定性

**做什么**：
1. 部署 **Wan 2.2 14B**（Apache 2.0）
2. 用 E-00 扩充数据训练 Character LoRA（高/低噪两个 expert）
3. 生成 10 段 5 秒视频，prompt 涵盖：
   - 静态站立（最简单）
   - 走路
   - 转身（视角变化）
   - 简单动作（挥手、坐下）
4. 关键评测维度：
   - **角色一致性**：每段视频抽 5 帧，CLIP-I 距离矩阵
   - **跨视频一致性**：10 段视频之间的 CLIP-I 距离
   - **画风稳定性**：aesthetic predictor 分布
   - **动作合理性**：DWpose 提骨看是否平滑、解剖正确
   - **手部 / 面部失真率**（视频生成的传统硬伤）
   - **VRAM、时延**

**报告关键点**：
- 当前视频生成距离"动作工厂"还差多远
- 哪些 prompt 类型表现好 / 差
- 角色 LoRA 在视频模型里是否真的起作用

---

### E-12: HunyuanVideo 同样测试
**做什么**：与 E-11 平行，但用 HunyuanVideo

**报告**：横向对比，谁更适合做我们的"动作工厂"
