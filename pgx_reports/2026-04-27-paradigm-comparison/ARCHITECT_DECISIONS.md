## TL;DR — recommend collapsing AC-0 to Path C (hybrid)

**Recommendation**: AC-0 → **Path C: 3D-identity + 2D-video-animation**.

```
Identity Asset (immutable, versioned) = 3D mesh
   ├─ generated by Hunyuan3D 2.0 (measured today: 263s on this hardware)
   ├─ retopped by trimesh quadric (measured: <1s)
   └─ rigged by UniRig on x86 worker (NOT measurable on aarch64 today)

Animation Asset (immutable, versioned) = 2D video clip
   ├─ Wan 2.2 I2V or Animate-14B (measured today: 23-25 min per clip on this hardware)
   ├─ conditioned on a 3D-rendered reference image of the composed character
   └─ stable identity (CLIP-I 0.89-0.95) when ref is honored

Online runtime = composes from cache; cache key = (identity × equipment × state × motion)
```

**Why Path C wins on the measurements**:
- Path A is **fast for identity** (4 min from image to textured mesh) but **rigging-incomplete on ARM**
- Path B is **fast for animation** (25 min for one motion clip) but **broken for equipment combinatorics** (CLIP-I drop from 0.95 → 0.61-0.75 on prompt-only swap)
- Combined: each side does what it's good at; together AC-3 is achievable

## Three concrete things to do this week (HIGH confidence — all measured)

### 1. Allocate one x86 dev machine for the rigging line

**Measurement supporting this**: UniRig requires `bpy==4.2`, `open3d`, `spconv-cu120`. Tested today on this aarch64 box: **all three lack Linux ARM wheels, period**. Source-build attempts: hit `libGL`/`libharfbuzz`/etc. dependency walls without sudo.

**On x86**: trivial install. UniRig README's standard `pip install` recipe works.

**Action**: spec a dedicated x86 rigging worker (modest GPU, e.g., RTX 4090 24GB enough). Budget: 1 day setup. Then it's a service.

### 2. Productionize Wan 2.2 Animate-14B as the animation generator

**Measurement supporting this**: ran end-to-end today. 25 min for 3.5 sec animated clip with **real motion** (CLIP-I 0.893 to ref, std 0.025 — meaningful per-frame variation = real motion).

**Pipeline**: ref image (3D-rendered or 2D) + driving video clip → animated video. Caching layer: `key = hash(ref + driving + seed)`.

**Action**: API service that takes (ref_image, driving_video) → MP4. Estimate 1-2 weeks engineering.

### 3. Productionize Hunyuan3D 2.0 as the identity / equipment generator

**Measurement supporting this**: 263 s on this hardware for a textured mesh. Custom CUDA ops compiled clean on aarch64+sm_120.

**Action**: queue runner (`hunyuan3d_jobs.py`) that takes images → glb. Half-day work. Now this hardware does both Wan 2.2 and Hunyuan3D — productionize both as services.

## The keystone task: Asset Contract design (HIGH confidence)

The 3 production lines that survive in Path C:

```yaml
IdentityAsset:
  uuid: UUID v7
  type: identity
  3d_mesh: <glb_path>
  rigged: optional<fbx_path>   # null until x86 rigger lands
  default_pose_render: <png_path>  # what we feed to Wan as ref

EquipmentAsset:
  uuid: UUID v7
  type: equipment
  category: enum {hair, top, bottom, shoes, accessory}
  3d_layer: <glb_path>          # mesh that overlays IdentityAsset
  texture: optional<png_path>
  bone_attachment: optional<bone_name>

AnimationAsset:
  uuid: UUID v7
  type: animation
  source_identity: IdentityAsset.uuid
  source_equipment: list<EquipmentAsset.uuid>
  source_motion: MotionAsset.uuid   # Mixamo offline FBX or driving video hash
  output: <mp4_path>
  cache_key: hash of source-tuple
```

One day of design + 0.5 day of implementation. **This unblocks everything else.**

## What I am still uncertain about

1. **Does the 3D→2D ref-render actually align stylistically with Wan 2.2?**
   We didn't run that test today. The Hunyuan3D PBR render style may differ enough from Wan 2.2's anime priors that the ref drifts heavily. **Risk-of-Path-C #1.**
   *Mitigation*: NPR pre-pass on 3D ref via Blender x86 worker. Untested but well-known pattern.

2. **Does Wan 2.2 Animate handle anime DRIVING videos (not just real-human videos)?**
   Today's run used Wan's bundled real-human dance video. Whether the preprocess can extract pose from anime motion (e.g., MMD) is unmeasured. **Risk-of-Path-C #2.**
   *Mitigation*: source 3-5 anime-style driving videos (open-source MMD) and re-test in next session.

3. **What's the real per-MP4-cache memory footprint at scale?**
   Each 25-min animation gen produces a 2.4 MB MP4. At thousands of (id × equip × motion) combos, the cache grows fast. **Risk-of-Path-C #3.**
   *Mitigation*: design Asset Repository with content-addressed storage from day 1.
