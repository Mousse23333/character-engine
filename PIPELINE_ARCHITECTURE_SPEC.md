## 1. 项目目标

构建一套**以 AI 为主力生产引擎、以传统渲染为主力消费引擎**的动态二次元 3D 角色管线。

最终目标是支持类似**绝区零 / 原神**视觉品质、**沙盒式可扩展**、**装备 × 状态空间爆炸**、
**实时合成**的角色系统。

下游消费者：现有 PIXI 2D 游戏（KinkiestDungeon mod），通过 3D→2D 投影桥接。

---

## 2. 范式：架构师 + AI 工人

| 角色 | 职责 |
|------|------|
| **人类（架构师）** | 决定系统拓扑、SLO、组件契约、组合规则；评判质量；处理异常样本 |
| **AI（工人）** | 执行所有可被规约的劳动密集环节：建模、绑骨、贴图、动画、shader、贴图变体 |
| **明确不学的事** | Blender / Maya / Substance / ZBrush 等传统 DCC 工具的**手艺层**操作 |
| **明确要学的事** | SLO 设计、Contract 设计、Workflow 编排、自动评测、AI orchestration |

核心信念：
- AI 在"人类劳动密集 + 低创造性判断"环节，**已经追平业余、逼近专家**
- AI 在"机器擅长的确定性计算"环节（GPU 渲染、物理、合成），**远不如专门算法**
- 所以：**AI 替代人，不替代机器**

---

## 3. 架构铁律（Architectural Constraints）

这些不是偏好，是后续所有设计必须满足的不变量。

| ID | 约束 | 含义 |
|----|------|------|
| **AC-1** | 3D 优先，可 3D→2D 渲染，纯 2D 排除 | 角色本体必须是 3D mesh；可在最后投影到 2D 喂给 PIXI |
| **AC-2** | 沙盒式管线 | 用户/玩家可扩展资产库；管线本身要可被运行时调用 |
| **AC-3** | 装备 × 状态空间爆炸 → 必须实时合成 | 不可能预生成所有组合；运行时合成是核心要求 |
| **AC-4** | 画风与角色解耦 | Style 与 Identity 是两个独立的资产维度 |
| **AC-5** | 可动化（Animation）是 open problem | 该环节当前业界开源解最弱，作为长期研究方向 |
| **AC-6** | 全程开源、可控、可改、可替换 | 每个组件必须有开源 fallback；不被任何闭源 SaaS 绑定 |
| **AC-7** | 离线生产 / 在线合成 严格分离 | 两侧只通过 Asset Contract 通讯；离线产物不可变、版本化；在线侧只读不写 |

---

## 4. 杂交路线（Hybrid Route）

```
原则 1：渲染 = 传统（GPU shader / 物理模拟 / 合成器）
原则 2：资产生产 = AI（mesh / 贴图 / 绑定 / 动画 / 风格）
原则 3：架构控制 = 人类（SLO / contract / 组合规则）
```

| 环节 | 谁来做 | 备注 |
|------|--------|------|
| 概念设计 | AI（Flux / Illustrious） | 已成熟 |
| 多视图概念图 | AI（多视图扩散 / Flux Kontext） | 已成熟 |
| 3D 雕刻 / mesh 生成 | AI（Hunyuan3D 2.0 / TRELLIS） | 已成熟（Apache 2.0 开源） |
| Retopology | AI 半自动（Instant Meshes 等） | 待评估开源选项 |
| UV 展开 | AI 半自动（Smart UV / 或自训） | 待评估 |
| PBR 贴图 | AI（Hunyuan3D-Paint / Substance + AI） | 已成熟 |
| 骨骼绑定 | AI（UniRig / 或工程化 ASMR） | AC-5/AC-6 红旗：开源工程化弱 |
| 蒙皮权重 | AI 半自动 | 同上 |
| 表情 blendshape | AI 半自动（ReveRig 类） | 已成熟 |
| 动画关键帧 | AI（视频→动捕 / Wan 2.2 + DWpose） | 待验证质量 |
| Shader 编写 | AI（LLM 写 GLSL / WGSL） | 已成熟（LLM 强项） |
| 装备建模 | AI（同 mesh 生成） | 已成熟 |
| 装备适配 / 防穿模 | **未解决**（业界硬伤） | **我们的研究问题之一** |
| **实时渲染** | **传统 GPU**（Three.js / Babylon / 自写 shader） | **AI 不应替代** |
| **物理模拟（布料/头发）** | **传统**（Magica Cloth / spring bone 类算法） | **AI 不应替代** |
| **合成（装备叠加 / 状态调制）** | **传统**（确定性程序合成） | **AI 不应替代** |
| 质量评判 | AI（CLIP-I / DINOv2 / Aesthetic / LLM-as-Judge） | 已成熟 |
| 失败样本 review | 人类 | 不可替代 |

## 5. 离线 / 在线分离

### 5.1 两个世界的根本差异

```
┌──────────────────────────────────────────────────────────┐
│           离线世界 (Offline / Production)                │
│  本质：把"概念"变成"可被消费的稳定数字资产"             │
│  时间尺度：小时—天                                        │
│  发生频率：稀疏                                          │
│  失败容忍度：高（可重跑、可人审）                         │
│  优化目标：质量、一致性、可复现性                         │
│  产出形态：不可变资产（immutable, versioned）             │
│  AI 占比：极高（核心生产引擎）                            │
└──────────────────────────────────────────────────────────┘
                          │
                          │  ★ Asset Contract（资产契约）
                          ▼   单向、不可变、版本化
┌──────────────────────────────────────────────────────────┐
│           在线世界 (Online / Composition)                │
│  本质：把"已有资产 + 当前请求"实时合成成"画面"           │
│  时间尺度：毫秒—秒                                        │
│  发生频率：高频                                          │
│  失败容忍度：低（必须 fallback）                          │
│  优化目标：延迟、确定性、可组合性                         │
│  产出形态：易变像素（ephemeral pixels）                   │
│  AI 占比：零（明确决定不留 AI 异常路径）                  │
└──────────────────────────────────────────────────────────┘
```

### 5.2 离线侧结构（7 条产线，独立可换）

```
┌──────────────────────────────────────────────────┐
│              离线世界 (Offline)                   │
│                                                   │
│  ① 风格指纹工厂  → Style Asset                    │
│     （训练 Style LoRA，学画风不学角色）            │
│                                                   │
│  ② 身份指纹工厂  → Identity Asset                 │
│     （训练 Character LoRA / IP-Adapter，学角色不学画风）│
│                                                   │
│  ③ Mesh 工厂     → Body Asset                     │
│     （Hunyuan3D / TRELLIS 出 3D mesh + 贴图）     │
│                                                   │
│  ④ 绑定工厂      → Rig Asset                      │
│     （骨骼 + 蒙皮权重；当前业界开源最弱环节）      │
│                                                   │
│  ⑤ 装备工厂      → Equipment Asset                │
│     （装备 mesh + skinned to character bones）    │
│                                                   │
│  ⑥ 状态调制工厂  → Modulator Asset                │
│     （受伤纹理 / 兴奋表情 / 装备破损 等的调制器）   │
│                                                   │
│  ⑦ 动作工厂      → Animation Asset                │
│     （视频生成 + 动捕提取 + 动作库）               │
│     （AC-5：长期 open problem，留长期演进口）     │
│                                                   │
│              ↓                                    │
│         自动质量评判（AI 评 AI）                  │
│         + 人工 review 不合格队列                  │
│              ↓                                    │
│        Asset Repository（不可变 + 版本化）        │
└──────────────────────────────────────────────────┘
```

**离线侧的特征**：
- 7 条产线**互相独立**，每条都可独立替换实现
- 全部产物**汇入同一个 Asset Repository**
- Repository **不可变 + 版本化**（v1 永远存在，v2 是新资产）
- 在线侧通过 Contract **只读**，不能反向写入

### 5.3 在线侧结构

```
┌──────────────────────────────────────────────────┐
│              在线世界 (Online)                    │
│                                                   │
│  请求输入：                                       │
│    (identity_id, style_id, equipment_set,        │
│     state_vector, view_angle)                    │
│           ↓                                       │
│  缓存层（多级）                                   │
│    - 命中：直接返回                               │
│    - 未命中：进入合成                             │
│           ↓                                       │
│  合成器（按层叠加，全程确定性）                    │
│    Body + Rig                                    │
│      ↓                                            │
│    + Equipment 叠加（mesh skinning + 冲突解决）   │
│      ↓                                            │
│    + State 调制（纹理叠加 / 形变 / 显隐）         │
│      ↓                                            │
│    + Pose 驱动（从 Animation Asset 取一帧）       │
│      ↓                                            │
│    + Style Shader 渲染（NPR 着色）                │
│      ↓                                            │
│    + 视角投影（3D → 2D）                          │
│           ↓                                       │
│  缓存写回 + 输出                                  │
│           ↓                                       │
│       游戏（PIXI 渲染）                           │
└──────────────────────────────────────────────────┘
```

**在线侧的特征**：
- 只读资产，不生产
- 全程**确定性合成**（同请求 → 同结果，无随机性）
- 优化重点：延迟、缓存命中率
- 失败必须有 fallback（用上一帧 / 基础姿态 / 占位贴图）
- **明确不留 AI 异常路径**（项目决策）

## 6. Asset Contract（最关键设计点 — 待详细设计）

整个架构的**单一脆弱点**。它定义了离线与在线之间的协议。

```
Asset Contract 的本质要求：
  - 不可变性：一旦发布，永不修改
  - 版本化：v1, v2, v3 共存
  - 自描述：资产携带元数据，在线侧不需要"知识"，只需要"读取"
  - 类型清晰：Style / Identity / Body / Rig / Equipment / Modulator / Animation 各自的 schema
  - 可组合：资产之间的组合规则在 contract 里说清楚
```

**当前未决**：每类资产的具体 schema、组合协议的数学/工程形式、版本兼容策略。

**这是下阶段最重要的工作**，比工具选型重要 100 倍。

## 7. 视频生成在管线中的位置

**决策**：视频生成只用于离线侧，不用于在线侧。

**用途**：
1. **动作工厂的核心实现路径**（产线 ⑦）
   - 用角色指纹 + 文本/姿态描述 → 生成参考视频
   - 用 DWpose / EasyMocap / SMPL-X 从视频提取动作
   - 重定向到角色骨骼 → 入动作库
2. **离线参考素材生成**
   - 生成参考视频供绑骨/动画/装备适配验证用

**待验证假设**：
- 当前 SOTA 视频模型（Wan 2.2 / HunyuanVideo / CogVideoX）的稳定性是否够用
- 角色 LoRA + 视频模型的"角色保真度"是否能满足下游要求
- 从生成视频提取动作的质量是否优于直接用 Mixamo 库

## 10. 当前未决问题（按优先级）

### 高优先级（架构级，必须先解决）

8. **AI Orchestrator 的实现形式** — Python + ComfyUI + LLM 编排？

### 低优先级（实现级，到细节阶段再说）
9. 具体的工具版本、shader 实现、PIXI 集成代码

### 任务 T1：视频生成稳定性测试（最高优先级）
**目的**：验证 H2 — 视频生成 + 角色 LoRA 的稳定性与还原度

**步骤**：
1. 选 **Wan 2.2 14B** 与 **HunyuanVideo** 两个模型本地部署
2. 使用上述测试角色（`test_character_alice_ref.jpeg`），按你选定的扩充策略产出 30-50 张训练数据
3. 训一个 Character LoRA（高/低噪两个 expert，参见 Wan 2.2 MoE 架构）
4. 用同一个 prompt（例：character_token + "walking forward" + 风格 tag）

6. 评判：
   - **角色一致性**：CLIP-I 跨视频比较；DINOv2 face embedding 距离

   - **动作合理性**：DWpose 提骨，看是否平滑、有无解剖错误

**报告格式**：每个维度给一组数字 + 你主观评级（"业余可用 / 接近专家 / 不可用"）+ 失败案例代表性截图

3. 喂给 Hunyuan3D 2.0 出 mesh + 贴图
4. 用 **Instant Meshes**（开源）做 retop
5. 用 **AccuRIG 2** 与 **UniRig**（开源）分别尝试自动绑骨
6. 用 Mixamo 标准动画驱动绑好的角色

**报告**：
- 原始 mesh 三角面数、retop 后面数、UV 质量评级
- 两种绑骨方案的对比（艺术质量、形变 artifact）
- 标准动画在该 mesh 上的播放是否正常
- 失败模式（哪类几何特征容易爆）

### 任务 T3：风格/身份解耦实验
**目的**：验证 H5 — 双 LoRA 叠加是否互相污染

**步骤**：
1. 训一个 Style LoRA（用 200+ 张同画风但**不同角色**的图）
2. 训一个 Identity LoRA（用 30+ 张**同角色不同画风**的图）
3. 四组对比生成：
   - a) 仅 Style
   - b) 仅 Identity
   - c) Style + Identity 叠加
   - d) 不带 LoRA
4. 评判：
   - c 应该同时呈现 a 的画风 + b 的角色身份
   - 测量 c 相对于 a 的"画风距离"、相对于 b 的"身份距离"
   - 报告污染程度（c 偏离哪边更多？）

### 任务 T4：硬件极限摸底
**目的**：摸清 128GB Blackwell 的并行上限

**步骤**：
1. 同时启动以下任务，记录 VRAM / 帧率 / 是否 OOM：
   - Hunyuan3D 推理 × 4 实例
   - Flux 推理 × 2 实例
   - Wan 2.2 LoRA 训练 × 1
   - ComfyUI 工作流 × 1
2. 寻找"性能崩溃点"（哪个组合开始变慢）

**报告**：可持续并行的最大组合、哪些任务互斥、夜间批处理的最佳调度建议

### 任务 T5：方向性反馈（最重要）
